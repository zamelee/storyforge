## 待开发 — Phase 25: 地理系统重构 + 重要地点

> 来源：社区反馈 + 产品梳理（2026-05-27）
> 优先级：高（现有结构有冗余和 bug）

### 问题

1. **自然环境 vs 地理环境重叠**：自然环境已有世界结构/大陆分布/山川河流/气候等，地理环境面板功能定位模糊
2. **两套地图功能冲突**：地理环境的「AI概念地图」与世界地图的 Voronoi 引擎重叠，用户不知道用哪个
3. **世界地图双主世界 bug**：`ensureRootWorld` 存在竞态条件，React StrictMode 下产生两个根世界
4. **多世界/诸天流方案未设计**：世界树只是简单的树结构，不读世界观数据，无法支撑诸天流等多世界观设定

### 计划

- **25.1** 修复世界地图双主世界 bug（`ensureRootWorld` 加锁/幂等）
- **25.2** 删除「地理环境」面板，地理总述合并到自然环境
- **25.3** 创作区新增「重要地点」模块
  - 地点类型 = 自然地形标签 + 人文场所标签（多标签组合，非下拉）
  - 自然地形：大陆、半岛、岛屿、群岛、高原、平原、盆地、丘陵、峡谷、山脉、山峰、火山、戈壁、沙漠、冰原、草原、森林、雨林、沼泽、绿洲、洞穴、海洋、海峡、海湾、湖泊、河流、瀑布、温泉、冰川、浮空岛、虚空、异界裂隙等
  - 人文场所：村庄、城镇、城市、都城、部落、营地、关隘、要塞、军营、战场、神殿、寺庙、学院、集市、酒楼、拍卖行、黑市、矿场、港口、驿站、废墟、遗迹、古墓、迷宫、禁地、秘境、宗门、洞府、灵脉等
  - 标签可组合（如「海岛 + 集市」=海岛上的集市；「海洋 + 海岛」=海洋中的海岛）
  - 支持树状父子层级（海洋 → 海岛 → 集市）
  - 三种视图：树状图、列表、（未来可接入世界地图标注）
- **25.4** 多世界方案设计文档（仅设计，不写代码；想清楚诸天流的数据结构再动手）

## 待开发 — Phase 26: 角色权重 + 逆向创作流程

> 来源：社区反馈（2026-05-27）
> 优先级：中-高（多位用户反馈角色与大纲/故事脱节）

### 问题

1. **创作流程断裂**：当前系统设计为「世界观→故事→角色→大纲→章节」自上而下，但多位用户反馈习惯先想人物再推剧情，生成的角色「不知道咋融合到故事里」
2. **大纲不读角色**：`buildVolumeOutlinePrompt` 和 `buildChapterOutlinePrompt` 完全不注入角色数据，用户先建了主角但大纲生成时 AI 不知道
3. **角色创建默认配角**：手动新建角色硬编码 `role: 'supporting'`，用户不知道怎么创建主角；AI 生成角色时也没有感知「当前缺什么角色」
4. **角色注入无权重**：`buildCharacterContext` 把所有角色平铺成一行文字，主角和配角信息密度一样，AI 无法区分重要性

### 用户故事

- **US-26A**：作为作者，我希望新建角色时能直接选择角色定位（主角/反派/配角…），而不是默认都是配角
- **US-26B**：作为作者，我希望 AI 生成角色时能感知当前阵容缺口（如「还没有主角」「缺一个反派」），优先补齐关键角色
- **US-26C**：作为作者，我希望生成大纲时 AI 能读取我已创建的角色，围绕主角展开主线
- **US-26D**：作为作者，我希望先设计好人物的初始状态和结局状态，让 AI 根据起终点生成中间剧情
- **US-26E**：作为作者，我希望写几句灵感/想法，AI 就能反向生成世界观、角色、故事核心

### 计划

- **26.1** 角色创建改进（快速修复）
  - 手动创角色时弹出 role 选择器（主角/反派/配角/次要/NPC/路人）
  - AI 生成角色时 prompt 注入阵容缺口信息（「当前：主角 0，反派 0，配角 2」）
- **26.2** 角色上下文分权重注入
  - `buildCharacterContext` 改为分层输出：核心角色（主角/反派，完整信息）→ 重要配角（一句话+关系）→ 其他（仅名字）
  - 大纲生成（`OutlinePanel`）注入角色上下文
  - 细纲、场景生成已有角色注入，但需对齐权重格式
- **26.3** 角色驱动剧情模式
  - 用户设定角色的「初始状态」和「目标状态/结局」
  - AI 根据两端状态 + 世界观约束，生成中间情节/大纲
  - 与现有大纲系统集成：生成结果可导入为卷/章结构
- **26.4** 灵感反推入口
  - 项目概况或独立面板，用户写碎片想法
  - AI 反向生成：世界观草稿、故事核心、初始角色卡
  - 生成结果填入对应模块，用户可二次编辑

## 待开发 — Phase 28: 作品分析结果优化

> 来源：社区反馈（2026-05-27）—— 用户反馈拆书结果散乱、重复、缺乏结构
> 优先级：中
> 依赖：现有作品学习系统（Layer 1+2）、导入分块分析

### 问题

1. **角色分析重复严重**：分块分析时同一角色在多个 chunk 中被重复分析（如截图中李凡、道玄子、寇洪各出现 3+ 次），用户看到 54 条角色分析但大量重复
2. **分析结果平铺无结构**：461 条大纲分析、12 条写作技法、12 条世界观全部平铺展示，没有目录导航，找不到具体内容在哪
3. **缺少总结环节**：分块分析各自独立，没有跨块汇总去重。如叙事视角与手法在每个事件中都不同，但平铺后失去了「哪个手法用在哪个场景」的对应关系
4. **大纲缺少分卷**：长篇作品的大纲分析有几百条，没有按卷/弧分组，无法快速定位
5. **文笔风格分析碎片化**：每个 chunk 各出一段文笔分析，内容高度重复且缺乏对比性

### 用户故事

- **US-28A**：作为作者，我希望拆书后的角色分析能按角色去重合并，每个角色一张卡片，汇总其在全书中的表现
- **US-28B**：作为作者，我希望分析结果页有侧边目录，按维度/章节快速跳转，而不是一条条翻
- **US-28C**：作为作者，我希望拆书完成后有一个「AI 总结」环节，将各 chunk 的碎片分析合并为精炼的全书总结
- **US-28D**：作为作者，我希望大纲分析能按卷分组，导入时支持手动分卷或让 AI 自动识别卷结构
- **US-28E**：作为作者，我希望写作技巧和叙事手法的分析能标注出处（第几章、什么场景），而不是脱离原文直接评价——光说「善用伏笔」没用，得告诉我在哪里用的、怎么用的
- **US-28F**：作为作者，我希望世界观分析有分区（地理/势力/规则等），不要所有内容杂糅在一起
- **US-28G**：作为作者，我希望每个功能/模块旁边有个小提示图标（如 ℹ️），点击能看到简要说明，降低上手门槛

### 计划

- **28.1** 分析结果去重、合并与出处定位
  - 角色维度：按角色名聚合，合并多个 chunk 对同一角色的分析，输出一张综合角色档案卡
  - 写作技巧/叙事手法维度：每条分析必须关联 chunk 来源（对应原文的章节范围），展示时标注「第 X-Y 章」，让用户能回溯到具体段落学习
  - 文笔风格维度：合并相似描述，保留差异化部分，附带典型例句出处
  - 世界观维度：按子分区（地理/势力/规则/文化等）分组展示，避免杂糅
  - 实现：分析完成后跑一轮本地合并（无需 AI），或用 AI 做跨 chunk 总结（可选）
- **28.2** 分析结果结构化展示
  - 侧边目录导航（按维度分组 + 锚点跳转）
  - 大纲分析按卷分组展示（识别卷标题/卷分隔）
  - 角色分析按角色卡片展示（去重后）
- **28.3** 全书总结环节
  - 所有 chunk 分析完成后，自动/手动触发「AI 全书总结」
  - 每个维度生成一段 200-500 字的精炼总结，替代碎片化的 chunk 级结果
  - 总结结果存入独立字段，展示在分析详情页顶部
- **28.4** 导入分卷支持
  - 导入长篇时识别卷标题（如「第一卷 xxx」「卷一」等），自动分卷
  - 支持手动拖拽分卷（在章节列表上划分卷边界）
  - 分卷信息在大纲分析中用于分组展示

### 与其他 Phase 的关联

- **与 Phase 26（角色权重）**：28.1 的角色去重合并为 26.2 的权重注入提供更干净的角色数据源——学习来的角色特征可以反哺到创作中
- **与现有大师洞察系统**：28.3 的全书总结可以直接作为「大师洞察」注入创作 prompt，比当前碎片化的 chunk 分析更有效
- **与 Phase 27（Agent）**：28.3 的 AI 总结是单轮调用就能做的，不需要等 Agent 架构；但 28.1 的跨 chunk 智能合并如果要做得更好，未来可升级为 Agent 多步推理

## 待开发 — Phase 27: AI Agent 化（历史考证 + NPC 自动演进）

> 来源：社区反馈（2026-05-27）—— 用户希望 AI 在构思时主动联想历史背景
> 优先级：低（需要 tool calling / agent 架构重构）

### 问题

1. 当前 AI 调用是单轮流式聊天，无法多步推理或调用外部工具
2. 历史题材用户希望：构思场景时 AI 主动搜索并联想符合历史背景的点子（需要搜索整合 + 上下文抽取，token 消耗大）
3. NPC 自动演进：主角遇到的 NPC，AI 在后台自动推演其故事线，在未来某章突然跳出来提供灵感（需要后台异步 agent）

### 计划（远期）

- **27.1** 架构评估：当前流式聊天 → 支持 tool calling 的 agent 模式，评估重构成本
- **27.2** 历史考证助手：构思场景时 AI 自动检索世界观中的历史设定 + 联想相关史实
- **27.3** NPC 自动演进：后台异步推演 NPC 故事线，在合适时机向作者推荐

## 待开发 — Phase 31: 历史模式全流程贯通

> 来源：内部代码审查（2026-05-27）—— 历史考证模式仅影响世界观生成，未贯通到大纲/细纲/正文
> 优先级：中-高（PR #9 的历史功能已合并但未串联到创作流程）

### 问题

1. **creativeMode 仅在世界观面板生效**：`project.creativeMode` 只被 `WorldviewOriginPanel` / `NaturalPanel` / `HumanityPanel` 读取，大纲/细纲/正文生成完全不感知
2. **历史年表数据未注入 AI 上下文**：`HistoricalTimelineEvent` 和 `HistoricalKeyword` 表有数据，但 `buildWorldContext` 不读取
3. **历史题材包与 creativeMode 开关脱节**：用户需要手动去提示词库切换历史题材包，creativeMode 切换不联动
4. **大纲不注入历史背景**：`buildVolumeOutlinePrompt` / `buildChapterOutlinePrompt` 无 creativeMode 参数

### 计划

- **31.1** 上下文注入历史数据
  - `buildWorldContext` 增加可选参数：读取 `historicalTimelineEvents`（按年份排序取关键事件）+ `historicalKeywords`（按分类分组）
  - 注入格式：`【历史时间线】开元十三年(725)：主角在长安开设织布机坊（虚构）...`
  - Token 预算控制：历史数据最多占上下文窗口的 10%
- **31.2** 大纲/细纲/正文感知历史模式
  - `buildVolumeOutlinePrompt` / `buildChapterOutlinePrompt` 新增 `creativeMode` 参数
  - 历史模式下 system prompt 追加：「严格遵循历史背景设定，标注架空与史实的边界」
  - 正文生成的 `CHAPTER_SYSTEM` 增加历史模式分支
- **31.3** creativeMode 联动题材包
  - 切换到「历史考证」模式时，自动提示用户是否同时切换题材包到「历史」
  - 切换到「幻想设定」模式时，同理提示是否切回通用/玄幻等
