# 设定层重构设计文档 — 词条系统 + 修炼体系

> Phase 35（世界观词条化重构）+ Phase 37（修炼体系）+ 关联 Phase 34（力量阶段追踪）/ Phase 36（上下游标记）
> 状态：**设计定稿**（4 项决策已拍板，修炼体系支持分叉）| 作者构想 + 实现设计
> 配套：`docs/ROADMAP.md`（Phase 34/35/36/37 概要）

## 〇、一句话目标

把现在"自由文本字段"式的世界观设定，重构为**词条（Codex）系统**——用户在分门别类的目录下创建一条条结构化词条（青金矿、白虎、某门派…），每类词条有专属字段；分类可由用户**自定义扩展**（多级大类/小类）。同时引入**修炼体系**（与世界底层力量体系分层），并让这套设定层与下游的「力量阶段追踪」「上下游标记」打通。

## 一、为什么要重构（现状问题）

| # | 问题 | 现状 |
|---|------|------|
| 1 | 道具重合 | 人文「道具与器物」字段 ↔ 侧栏「道具系统」面板，两套 |
| 2 | 自然/人工混淆 | 物产没分天然产出 vs 人工产出 |
| 3 | 不可词条化 | 都是大段自由文本，无法逐条结构化管理 |
| 4 | 活物归类不当 | `NaturalResources` 把珍禽异兽和矿石混在一个对象里 |
| 5 | 归属错误 | 自然环境的「重镇分布」其实是人文内容 |
| 6 | 堆叠太挤 | 人文「种族民族 / 政治经济文化」全堆一个字段链 |
| 7 | 分类僵化 | 预设分类覆盖不全，用户无法扩展 |
| 8 | 力量层级混淆 | 世界底层能量 与 修炼流派境界 没分开 |

## 三、词条系统（Codex）数据模型

### 3.1 核心：两张通用表 + 类型 schema

不为每类词条单独建表（否则自定义分类无处安放），而是用**通用词条表 + 分类树 + 字段 schema**：

```typescript
/** 词条分类（树状，内置 + 用户自定义） */
export interface CodexCategory {
  id?: number
  projectId: number
  /** 所属领域：自然 / 人文 */
  domain: 'natural' | 'humanity'
  /** 父分类 id（null = 顶层大类）；支持多级大类→小类 */
  parentId: number | null
  name: string
  icon?: string
  /** 内置分类的稳定标识（自定义分类为空）；用于代码识别内置类 */
  builtInKey?: string   // 如 'mineral' | 'herb' | 'beast' | 'race' | 'faction' | 'city' | 'artifact'
  /** 该分类下词条的字段 schema（字段定义数组，JSON） */
  fieldSchema: string   // CodexFieldDef[] 序列化
  order: number
  /** 多世界：所属世界组（null = 主世界/单世界） */
  worldGroupId?: number | null
  createdAt: number
  updatedAt: number
}

/** 字段定义 */
export interface CodexFieldDef {
  key: string           // 内部键
  label: string         // 显示名（外观/品级/功效…）
  type: 'text' | 'longtext' | 'select' | 'number' | 'ref'
  options?: string[]    // select 选项
  refCategory?: string  // ref 字段：指向哪类词条（如器物的"所需材料"→矿物/草药）
  required?: boolean
}

/** 一条词条 */
export interface CodexEntry {
  id?: number
  projectId: number
  categoryId: number    // 所属分类
  name: string
  icon?: string
  summary: string       // 一句话简介
  description: string   // 详细描述（自由文本）
  /** 按所属分类的 fieldSchema 填的结构化字段（JSON：{ [key]: value }） */
  fields: string
  /** 与其它词条的关联（JSON：{ [relKey]: entryId[] }，如器物→材料词条） */
  refs?: string
  order: number
  worldGroupId?: number | null
  createdAt: number
  updatedAt: number
}
```

**为什么这样设计**：
- 通用表 → 自定义分类天然支持（新分类 = 新 `CodexCategory` 记录 + 自定义 `fieldSchema`）
- 分类树（`parentId`）→ 满足"大类下加小类、自定义大类下再加小类"
- `fieldSchema` 驱动 UI → 不同类型词条自动渲染不同字段表单
- `refs` → 实现"矿物→可炼器物、器物→所需材料"的关联链
- 多世界 `worldGroupId` → 复用现有按世界隔离机制

### 3.2 内置分类与字段 schema（预置 seed）

> 通用字段（所有词条）：名称 / 图标 / 一句话简介 / 详细描述。以下为各类**专属字段**。

| 内置类 | builtInKey | domain | 专属字段 |
|--------|-----------|--------|---------|
| ⛏️ 矿物灵材 | `mineral` | natural | 外观(形状/颜色/质感) · 品级品阶 · 功效作用 · 产地分布 · 稀有度 · 可炼器物(ref) |
| 🌿 灵植草药 | `herb` | natural | 形态 · 药效 · 品级 · 生长环境 · 成熟周期 · 采集难度 · 可炼丹药(ref) |
| 🐅 灵兽异兽 | `beast` | natural | 类别(走兽/飞禽/水族/虫豸/异种) · **修炼体系(ref→修炼体系)** · 当前境界 · 体型外貌 · 习性 · 栖息地 · 威胁等级 · 特殊能力 · 可产出材料(ref) |
| 🧬 种族民族 | `race` | humanity | 外貌特征 · 种族天赋 · 平均寿命 · 人口规模 · 聚居地 · 文化习俗 · 信仰 · 与其他种族关系 · 代表人物 |
| ⚔️ 势力 | `faction` | humanity | 类型(门派/朝廷/商会/部落) · 势力范围 · 领导者 · 核心成员 · 实力等级 · 宗旨目标 · 敌友关系 · 标志旗帜 · **绑定地图区域**(合并 Faction.mapRegion) · 颜色 |
| 🏰 城池重镇 | `city` | humanity | 所属势力(ref→势力) · **地理位置(ref→重要地点)** · 规模人口 · 统治者 · 经济特产 · 战略地位 · 城市风貌 · 地标建筑 |
| 🗡️ 人工器物 | `artifact` | humanity | 类别(武器/防具/法器/丹药/功法秘籍/阵法/材料) · 品级品阶 · 外观 · 能力效果 · 炼制方式 · 所需材料(ref→矿物/草药) · 来历 · 当前持有者 |

### 3.3 自定义分类（用户确认必做）

- 用户可在自然/人文任意位置**新增自定义大类**（如人文下加「秘术组织」）
- 自定义大类下可再**加自定义小类**（多级）
- 自定义分类可**自定义字段 schema**（增删字段、选字段类型），或一键继承某内置类的 schema
- 内置分类不可删但可"隐藏"；用户自定义分类可删（连带词条，二次确认）
- 实现参考：现有 `world-rules` 的 `customNodes` 已验证过"内置 + 自定义节点树"模式，可借鉴

## 四、修炼体系（Phase 37）

### 4.1 与世界底层力量体系的分层

| | 世界底层「力量体系」（已有，保留） | 「修炼体系」（新增） |
|---|---|---|
| 是什么 | 世界**底层的能量本身** | 利用能量的**不同流派/方式** |
| 例 | 灵气 / 魔力 / 斗气 | 武夫境界、术士境界、召唤师境界… |
| 数量 | 一种（或少数） | **多个**，用户自建 |
| 存储 | 世界起源 `powerHierarchy` 字段（不动） | 新表 `cultivationSystems` |

### 4.2 数据模型

```typescript
/** 修炼体系（多套，用户自建） */
export interface CultivationSystem {
  id?: number
  projectId: number
  name: string          // 武夫 / 术士 / 召唤师 / 炼丹师…
  description: string
  /** 境界阶梯（JSON：CultivationStage[]，支持分叉/合流，构成 DAG） */
  stages: string
  worldGroupId?: number | null   // 多世界：每世界可有不同体系集合
  createdAt: number
  updatedAt: number
}

/**
 * 单个境界阶段。通过 parentStageIds 构成有向无环图（DAG），支持：
 *  - 线性：单链（炼体→易筋→洗髓）
 *  - 分叉：一阶 → 多个后续分支（金丹后分「剑修」「体修」「丹修」）
 *  - 合流：多个前置 → 同一后续（不同流派殊途同归到「大乘」）
 */
export interface CultivationStage {
  id: string
  name: string          // 炼体 / 金丹 / 剑婴(剑修分支)…
  features?: string     // 该境界的特征/战力描述
  breakthrough?: string // 突破到此阶的条件
  /** 前置阶段 id 列表（空 = 起始阶；多个 = 合流点） */
  parentStageIds: string[]
  /** 分支标签（同一分叉点的不同分支用，如「剑修」「体修」） */
  branchLabel?: string
  /** 可选：层级/深度，用于图形布局排序 */
  tier?: number
}
```

> **分叉设计要点**：阶段不再用 `order` 线性排序，而是用 `parentStageIds` 构成 DAG。起始阶 `parentStageIds=[]`；分叉 = 某阶被多个后续阶引用为父；合流 = 某阶有多个父。`branchLabel` 标注分支名，便于 UI 和力量追踪区分"主角走的是哪条路"。

### 4.3 关联

- **异兽词条**：`beast` 类的"修炼体系"字段 ref 到某 `CultivationSystem`，"当前境界"取其某一阶
- **角色**：Character 可标注主修的修炼体系（新增可选 `cultivationSystemId`），供 Phase 34 追踪参照
- **AI 上下文**：修炼体系定义属上游设定，写战斗/突破时注入，AI 遵守境界设定
- **多世界**：诸天流每个世界可有不同修炼体系集合

## 五、Phase 34 力量阶段追踪（下游产物，依赖 37）

- **参照修炼体系阶梯**（不是世界底层力量体系）：把角色主修的 `CultivationSystem.stages`（含分叉）喂给 AI
- AI 从已写正文检测主角当前境界 + **走的是哪条分支** → 记录到新表 `cultivationProgress`（角色 + 体系 + 当前阶 id + 已走路径 stageId[] + 每次突破章节/触发）
- 展示：当前境界 + 晋升历程时间线（"修为进度条"），分叉体系下高亮主角实际走过的那条路径
- 可选注入后续章节写作，防境界倒退、防走错分支
- 归属"下游提取产物"，与物品栏/故事年表同构（复用逐章提取 + 去重管线）

## 七、势力/阵营合并 & 历史线并入历史年表

### 7.1 势力 = 阵营（合并）

- 「势力词条」(`faction` 类) 取代独立 `Faction` 表与人文「势力分布」字段

- 人文不再有「世界历史线 / 世界大事记」独立字段

- 历史年表已支持多世界标签（Phase 25.5.1），天然兼容

## 八、数据迁移与导入（用户确认）

### 8.1 不做旧数据自动迁移

重构后**不强行迁移**用户已填的自由文本（`NaturalResources` / `itemDesign` / `factions` 等）。旧字段可保留只读"历史数据"区，或在重构上线说明中提示用户手动整理。

### 8.2 导入时 AI 介入分类（关键）

用户**主动选择「导入」**时，走导入流程，**AI 把导入内容按新分类分门别类填进对应词条**：
- 复用现有导入流水线（`import-adapter` / `import.parse-*` seed）
- 新增 prompt：把一段世界设定文本/文档，AI 识别其中的矿物、草药、异兽、种族、势力、城池、器物，分别结构化为对应类的词条
- 字段严格对齐各类 `fieldSchema`（延续"AI 解析不用正则 + 字段对齐"原则）
- 导入预览 → 用户勾选/修改 → 批量写入词条表

## 九、与现有代码的对应（落地映射）

| 重构对象 | 现有代码 | 处理 |
|---------|---------|------|

## 十、UI 设计

### 10.1 词条面板（通用）

```
┌─ 自然物产 / ⛏️ 矿物灵材 ───────────────[切世界]─┐
│  [+ 新建词条]  [✨ AI 生成]  [搜索]              │
│  ┌─ 词条列表（左）─┐ ┌─ 词条详情（右）──────┐ │
│  │ 青金矿          │ │ 名称 [青金矿]          │ │
│  │ 寒铁            │ │ 外观 [...]            │ │
│  │ + 自定义小类     │ │ 品级 [下品 ▾]         │ │
│  └────────────────┘ │ 功效 [...]            │ │
│  [+ 新增大类/小类]    │ 可炼器物 [关联→寒铁剑]  │ │
│                      └──────────────────────┘ │
└────────────────────────────────────────────────┘
```

- 左侧分类树（内置 + 自定义，可折叠），右侧词条列表 + 详情表单（由 fieldSchema 驱动）
- 每个字段支持单字段 AI 生成（复用现有模式）
- `ref` 字段渲染为关联选择器（选其它词条）
- 多世界顶部世界切换器

### 10.2 修炼体系面板（支持分叉）

- 列出多套修炼体系，每套可编辑名称 + **境界图谱**
- 境界图谱用节点图编辑（复用世界关系流向图 `WorldRelationGraph` 的 SVG/布局思路）：
  - 每个节点 = 一个境界（名称/特征/突破条件）
  - 连线 = 父子关系；从某阶拉出多条 = 分叉，多条汇入一阶 = 合流
  - 分支用 `branchLabel` 标注（剑修/体修…）
  - 自动按 `tier`/DAG 拓扑布局，纵向从低到高
- 简单体系（无分叉）退化为一条竖直链，编辑体验和线性一致
- 角色卡可下拉选择主修体系（力量追踪据此参照）

## 十二、风险与对策

| 风险 | 对策 |
|------|------|
| 大重构动核心世界观面板 | 分期：先地基（新表/面板）与旧并存，再逐类切换、最后下线旧入口 |
| 通用词条 schema 复杂度 | fieldSchema 驱动表单，内置类预置；自定义类继承模板降低门槛 |
| 势力合并影响角色关系引用 | 迁移期保留 Faction 读路径兼容，关系引用平滑切换 |
| 城池 vs 重要地点重叠 | 设计阶段确认二者关系（合并或 city 词条 link 到 location） |
| 自定义分类 + 多世界 + 导入交叉复杂 | 每个维度独立验证后再组合；导入预览必经用户确认 |
| 旧数据 | 不自动迁移，降低风险；导入路径补 AI 分类 |

## 十三、设计决策（已拍板 2026-06-03）

1. ✅ **城池重镇 link 到「重要地点」**：不再另起一套地点。城池词条的"地理位置"字段 ref 到现有 `ImportantLocation`（树状带标签的地点系统），城池词条聚焦"人文属性"（所属势力/统治者/特产/战略地位），地理坐标层面复用重要地点。
2. ✅ **宏观描述保留概述，实体才词条化**：政治/经济/文化/世界结构/疆域/气候/矛盾冲突等宏观描述保留为概述自由文本字段；只有"可枚举的实体"（矿物/草药/异兽/种族/势力/城池/器物）才词条化。
3. ✅ **角色卡关联种族词条**：Character 新增可选 `raceEntryId`，角色卡下拉选择所属种族（指向 `race` 词条）；同理可关联主修修炼体系 `cultivationSystemId`。
4. ✅ **修炼体系支持分叉**（用户确认必做）：阶段用 DAG（`parentStageIds`）支持分叉/合流（金丹后分剑修/体修、先天后天分流、殊途同归到大乘等）。见第四章数据模型 + 10.2 图谱编辑。力量追踪记录主角实际走过的分支路径。
