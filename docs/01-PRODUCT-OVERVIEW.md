# StoryForge / 故事熔炉 — 产品概述

> **版本**: v1.0 | **最后更新**: 2026-04-13 | **状态**: 规划中

---

## 1. 产品定义

**故事熔炉（StoryForge）** 是一款纯前端、开源的 AI 小说创作工坊。数据全部存储在用户浏览器本地，AI 模型由用户自由选择和配置（支持 DeepSeek / OpenAI / 通义千问 / 豆包 / Ollama 本地模型），零后端、零注册、零成本。

### 一句话定位

> "构建世界观 → AI 生成大纲 → AI 写章节正文 → 导出完整小说"

### 核心价值主张

| 用户痛点 | StoryForge 的解法 |
|---------|------------------|
| AI 直接写小说，前后不连贯、人设崩坏 | 完整的世界构建引擎，10 大维度构建世界观，自动注入上下文 |
| 商业工具收费贵，数据在别人服务器 | 纯前端开源，数据存本地 IndexedDB，自带 API Key |
| 只能用平台指定的模型 | 自由选择任何 OpenAI 兼容 API，包括 Ollama 本地模型（完全离线） |
| 大纲生成一次性，无法分批处理超长篇 | 树形大纲系统，支持卷→章节层级结构，分批生成 |
| AI 生成的文字"AI 味"太重 | 内置"老贼"写手人设 + 去 AI 味润色功能 |
| 伏笔管理全靠脑子记 | 10 种伏笔模式库 + 伏笔状态追踪（埋设→呼应→回收） |

## 2. 目标用户

### 主要用户画像

**画像 A：网文作者 / 小说爱好者**
- 有创作热情，想写长篇小说但缺乏经验
- 世界观设定散落在各种笔记/文档中，缺乏系统管理
- 尝试过用 ChatGPT 直接写小说，但质量不满意（前后矛盾、人设不一致）
- 对工具的需求："帮我管理好世界观和角色，让 AI 写的时候不会忘记设定"

**画像 B：AI 写作探索者**
- 对 AI 辅助创作感兴趣，喜欢尝试各种模型
- 不想被锁定在某个商业平台
- 可能在本地运行 Ollama 模型
- 对工具的需求："自由选择模型，数据在自己手里"

**画像 C：网文写手 / 日更作者**
- 日更 3000-5000 字，效率是第一需求
- 需要 AI 帮忙扩写/续写/润色
- 需要大纲管理和伏笔追踪
- 对工具的需求："帮我提高日更速度，不要让剧情出 bug"

### 非目标用户

- 不使用 AI 辅助的传统作者
- 追求文学性的纯文学作者（工具偏向网文/通俗小说）
- 只写短文/段子的用户（工具偏向长篇小说）

## 3. 竞品分析

### 直接竞品

| 竞品 | 优势 | 劣势 | StoryForge 的差异化 |
|------|------|------|-------------------|
| **在线AI写作平台** | 功能完整，世界构建体系成熟 | 闭源收费，数据在服务器，模型不可选 | 开源免费，数据本地，模型自选 |
| **NovelAI** | 二次元风格强，模型微调质量高 | 只有英文，按量收费，纯文字生成无世界观管理 | 中文原生，完整世界构建引擎 |
| **ChatGPT/Claude 直接写** | 最灵活，模型能力强 | 无上下文管理，无法维持长篇一致性，无大纲/伏笔系统 | 自动上下文注入，系统化创作流程 |
| **Sudowrite** | 英文写作辅助成熟 | 英文为主，收费，无世界观系统 | 中文原生，开源免费 |

### StoryForge 的独特壁垒

1. **纯前端 + 数据本地** — 零后端成本，隐私安全，可完全离线使用
2. **完整世界构建引擎** — 10 大维度系统化构建世界观
3. **模型自由** — 支持任何 OpenAI 兼容 API + Ollama 本地模型
4. **开源** — 商业竞品不会开源（会蚕食付费用户），这是护城河
5. **专业提示词系统** — 内置多种 AI 角色人设（世界设计师、大纲师、"老贼"写手等）

## 4. 产品形态

### 技术形态

- **纯前端应用**（无后端服务器）
- **单页应用（SPA）**，使用 React + TypeScript
- **静态部署**于 Vercel / GitHub Pages / Netlify
- **用户自带 API Key**（用于所有 AI 功能）
- **支持 Ollama 本地模型**（完全离线使用）
- **数据存储**：IndexedDB（通过 Dexie.js）

### 技术栈选型

| 层 | 技术 | 理由 |
|---|------|------|
| 框架 | React 18 + TypeScript | 与 yuntype/infiniteskill 保持一致 |
| 构建 | Vite | 快速，已有经验 |
| 状态管理 | Zustand | 轻量，类 Pinia 体验 |
| 本地数据库 | Dexie.js (IndexedDB) | 浏览器本地存储大量结构化数据 |
| 样式 | Tailwind CSS | 快速搭建暗色主题 UI |
| 富文本编辑 | TipTap / CodeMirror | 章节正文编辑器 |
| 部署 | Vercel | 已有部署经验 |

### 不需要 API Key 的功能

- 项目创建/管理
- 世界观设定手动录入
- 角色管理（手动创建/编辑）
- 大纲手动编辑
- 章节手动编写
- 伏笔管理
- 数据导入/导出（JSON 备份）
- 全文导出（Markdown/TXT）

### 需要 API Key 的功能

- AI 生成世界观（各维度）
- AI 生成角色设定
- AI 生成大纲
- AI 生成章节正文
- AI 续写/扩写/润色/去 AI 味
- AI 生成伏笔建议

## 5. 核心用户流程

### 创作全流程

```
[1] 创建项目
     ↓
[2] 基本信息（书名/类型/受众/核心设定/流派）
     ↓
[3] 世界构建引擎（10大维度）
    ├─ 世界观概述
    ├─ 故事核心（主题/冲突/情节模式）
    ├─ 角色管理（主角/配角/关系）
    ├─ 势力阵营
    ├─ 力量体系/功法体系
    ├─ 地理环境
    ├─ 历史年表
    ├─ 社会结构
    ├─ 道具/法宝系统
    └─ 规则约束
     ↓
[4] 大纲系统
    ├─ AI 生成整体大纲
    ├─ 树形结构：卷 → 章节
    ├─ 每个节点可编辑/AI重写
    └─ 分批生成（适合超长篇）
     ↓
[5] 章节写作
    ├─ 从大纲选择章节
    ├─ AI 生成正文（自动注入世界观+角色+大纲上下文）
    ├─ 续写/扩写/润色/去AI味
    └─ 手动编辑
     ↓
[6] 伏笔管理
    ├─ 埋设伏笔（关联章节）
    ├─ 呼应伏笔
    └─ 回收追踪
     ↓
[7] 导出
    ├─ Markdown
    ├─ TXT
    └─ JSON（完整项目备份）
```

### AI 配置流程

```
打开设置面板
     ↓
选择 AI 提供商
├─ DeepSeek（推荐，性价比高）
├─ OpenAI / ChatGPT
├─ 通义千问
├─ 豆包
├─ Ollama（本地模型，完全离线）
└─ 自定义 OpenAI 兼容端点
     ↓
输入 API Key（Ollama 无需 Key）
     ↓
选择模型
     ↓
测试连接
     ↓
保存（存 localStorage）
```

## 7. 设计原则

### UI 设计原则

1. **暗色主题优先** — 小说创作是长时间沉浸式工作，暗色主题减少视觉疲劳
2. **面板式布局** — 左侧导航 + 中间编辑区 + 右侧属性面板，类 IDE 布局
3. **信息密度适中** — 世界设定信息量大，需要合理的层级和折叠
4. **状态可视化** — 伏笔状态、章节完成度、字数统计等需要直观展示

### AI 交互原则

1. **流式输出** — 所有 AI 生成都使用 SSE 流式显示，实时可见
2. **可中断** — 用户随时可以停止 AI 生成
3. **可编辑** — AI 生成的所有内容都可以手动修改
4. **上下文透明** — 用户可以看到发给 AI 的完整上下文（可折叠查看）
5. **优雅降级** — 没有 API Key 时，所有功能仍可手动使用

### 数据原则

1. **本地优先** — 所有数据存浏览器 IndexedDB，不上传任何服务器
2. **可导出** — 随时可以导出完整项目数据（JSON）
3. **可导入** — 支持导入备份数据
4. **无锁定** — 导出格式开放，不锁定用户
