# 词条化重构施工计划（Phase 35-b/c）· 施工权威

> 创建 2026-06-09。本文件是「词条化重构」的**唯一施工权威**,任何接手者动手前必读。
> 设计依据：`docs/CODEX-REDESIGN.md`（数据模型与信息架构）。本文件是它的**分步施工蓝图**。
> 上一次词条化跑偏的教训：把"补充入口"做成了"唯一实现",旧面板/道具系统原封不动,造成三处重复。**本计划的全部意义是不再跑偏。**

---

## 0. 北极星（最终形态 · 不可偏离）

1. **词条化是"长在自然/人文面板里"的组织方式,不是独立模块。** 独立「设定词条」侧栏最多保留为只读聚合入口,不是唯一/主入口。
2. **每个概念全局唯一来源,零重复。** 不允许自然/人文概念同时存在于「世界观面板 + 设定词条 + 道具系统」。
3. **侧栏「道具系统」下线**,数据并入人文「人工器物」词条。
4. **词条功能本身**(区别于"放在哪")：
   - 结构化字段(由分类的 `fieldSchema` 驱动,不同类型不同表单)
   - **专属字段用户可自定义**(增/删/改名/调序/换类型,内置类也能改)——零 DB 迁移(改的是 fieldSchema JSON)
   - 词条**互相关联**(`refs`：矿物→可炼器物、器物→所需材料,"材料↔成品"链)
   - 关联其它系统(城池→重要地点、势力→地图区域、异兽→力量/修炼体系)
   - **词条进 AI 生成上下文**(`codex` 源被剧情/正文/大纲 need → AI 可调用用户造的素材,剧情更多样)

## 1. 范围红线（IN / OUT）

- ✅ IN：**整合**(词条收口进面板)、**消重**(删三处重复)、**下线**(道具系统)、**接上两个固有能力**(自定义字段 UI、进 AI 上下文)、自然资源纯文本→词条化。
- ❌ OUT：**不是"再加新功能"**。不新增与最终形态无关的东西。任何"顺手加个…"的念头 → 拒绝。
- 🚫 不强制迁移旧数据(用户决策)。但旧数据**不可丢**:保留展示 + 提供"一键转词条(AI 分类)"。

## 3. 施工铁律（每步都要守）

- **守三注册表**：新表/字段走 `PROJECT_TABLES`;词条写回走 `adopt()`;词条进上下文走 `CONTEXT_SOURCES`。
- **每步独立可验证、可单独回滚**：每个 Stage 一个分支 + 反例测试 + tsc/build 全绿才合并。
- **旧数据零丢失**：下线/迁移任何模块前,先确认旧数据有去处(并入或保留),写往返测试。
- **不删后补**：先迁移/兼容,确认无损,再下线旧入口。绝不先删后补。

## 4. 分步施工（按风险从低到高；每步：目标/改法/验证/完成判据/回滚）

### Stage B0 — 现状核实 + 内置分类 seed（零 UI 风险）
- **目标**：确认/补齐 7 个内置分类(矿物/草药/异兽/种族/势力/城池/器物)的默认 `fieldSchema` seed;确认 CodexPanel 现有能力(是否已支持自定义字段/ref)。
- **改法**：若未 seed,加幂等 seed(已存在不重复建,走 adopt/store);不动任何现有面板。
- **验证**：新建项目后 7 个内置分类带默认字段就位;反例测试断言 seed 幂等。
- **完成判据**：内置分类与字段 schema 齐备,旧面板零改动。
- **回滚**：删 seed 即可。

### Stage B1 — 词条编辑器完善：自定义字段 + ref 关联（在独立 CodexPanel 内先做）
- **目标**：词条卡片按 fieldSchema 渲染表单;支持**「管理字段」**(增删改 fieldSchema);支持 **ref 字段选择器**(材料↔成品)。
- **改法**：在现有 `CodexPanel` 内补齐(先不嵌面板,降低耦合);fieldSchema 改动走 store/adopt。
- **验证**：能给"矿物"加一个自定义字段并填值;能把"器物"的"所需材料"关联到某矿物词条;反例测试覆盖 fieldSchema 增删 + ref 存取。
- **完成判据**：词条系统功能完整(结构化+自定义+关联),仍在独立面板内。
- **回滚**：UI 改动,回退组件即可。

### Stage B2 — 词条进 AI 生成上下文（一行 need + 验证）
- **目标**：剧情/正文/大纲生成 `need: 'codex'`,AI 能调用词条素材。
- **改法**：核实 codex 源现有 `read`(`buildCodexContext`)输出质量;在相关生成入口 sourceKeys 补 `'codex'`(同 FB-9 手法)。
- **验证**：抓包/单测断言含词条内容进入生成 prompt。
- **完成判据**：词条内容确实进 AI 上下文。
- **回滚**：去掉 sourceKeys 那行。

### Stage B3 — 嵌入「自然环境」面板（收口 · 自然物产词条化）
- **目标**：自然环境面板 = 概述字段 + 内嵌「自然物产」词条区(矿物/草药/异兽);旧"自然资源"纯文本提供**兼容展示 + 一键转词条**。
- **改法**：面板内渲染该 domain 下的词条;旧纯文本字段保留(标"旧版,可转词条")。
- **验证**：自然环境面板能建/看自然物产词条;旧数据仍可见;往返测试不丢。
- **完成判据**：自然物产在自然环境面板内词条化,旧数据零丢。
- **风险**：动核心面板。**先在面板内"新增词条区",不删旧字段**。

### Stage B4 — 嵌入「人文环境」面板（收口）
- **目标**：人文面板内嵌 种族/势力/城池/器物 词条区 + 政治/经济/文化/矛盾冲突概述字段;「重镇分布」从自然移来。
- 改法/验证/判据同 B3 模式。

### Stage C1 — 下线「道具系统」（消重 · 高风险数据）
- **目标**：道具系统数据并入人文「人工器物」词条;侧栏 `items` 入口移除。
- **改法**：写迁移/兼容(旧 ItemSystem 数据 → 人工器物词条),**先迁移+往返测试确认无损,再移除侧栏**;`PROJECT_TABLES` 相应调整。
- **验证**：旧道具数据在人工器物词条里完整可见;导出导入往返不丢。
- **完成判据**：道具系统侧栏消失,数据全在人工器物词条,零丢失。
- **回滚**：保留旧表,仅隐藏入口;数据问题可回退。

### Stage C2 — 势力合并到「势力」词条（消重）✅ 已完成
- **目标**：「势力」词条合并原 `Faction` 表(含 mapRegion/color 地图绑定字段)。
- **已落地**：新增 `migrations/faction-to-codex`(幂等·自愈·先迁移后删);下线孤儿 `FactionPanel`
  (无导航入口的死路由)+ 清理 character store 的势力 state/方法;`factions` 表/导出导入保留兼容旧备份。
- **验证**：R-C2 回归(字段含 mapRegion/color 完整 / 旧行清空 / 幂等 / 空数据无害);119 测试全绿。
- **注**：原计划把「势力合并」与「历史线并入年表」捆在一个阶段——经确认拆开,历史线另立 Stage C2b。

### Stage C2b — 世界历史线并入历史年表（消重 · 待做）
- **目标**：人文 `worldview.historyLine`(文本)并入 `storyTimelineEvents`(逐条年表)。
- **改法**：需一次 AI 把历史线文本抽成结构化事件(走 assembleContext + adopt,**遵全局铁律:文本拆解用 AI 不用正则**),旧文本兼容保留。

### Stage C3 — 关联系统接线
- **目标**：城池词条「地理位置」ref 到「重要地点」;势力词条「绑定地图区域」与世界地图联动。

### Stage C4 — 独立「设定词条」侧栏降级（消重收尾）
- **目标**：独立 `CodexPanel` 降为**只读聚合入口**(跨自然/人文总览),或移除;主入口在面板内。
- **完成判据**：自然/人文概念**全局唯一来源**,不再三处重复。

### Stage D — 旧数据兼容收尾 + 导入 AI 分类
- **目标**：旧自然资源/道具纯文本「一键转词条」(AI 按新分类归类);导入流程 AI 按新分类入词条。
